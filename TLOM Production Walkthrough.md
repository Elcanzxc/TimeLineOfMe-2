Time Line Of Me — Как это работает в продакшне
Полный пошаговый сценарий работы платформы глазами пользователя и системы.

=== Этап 1: Лендинг — Первое впечатление (Анимированная история жизни) ===

При заходе на сайт пользователя встречает полноэкранная анимированная история,
показывающая путь человека через медиа и искусство на каждом этапе жизни.
Это не просто заставка — это эмоциональная демонстрация идеи проекта.

Сцена 1 — Детство:
  Маленький ребёнок сидит на полу и играет в простую игру (приставка/геймбой).
  Мама или папа читает ему сказку — книгу перед сном.
  Ребёнок сидит перед старым телевизором и смотрит мультфильм/фильм.
  Из того же телевизора звучит музыка — ребёнок слушает.
  → Рядом появляются карточки: «Том и Джерри», «Гарри Поттер», детские песни.

Сцена 2 — Юность (студент):
  Молодой человек сидит за ПК и играет в компьютерную игру.
  Надевает наушники — слушает музыку.
  Читает книгу (учебник или роман) за столом.
  Смотрит фильм на ноутбуке или телевизоре в комнате.
  → Карточки: «GTA», «Linkin Park», «Мастер и Маргарита», «Властелин Колец».

Сцена 3 — Взрослая жизнь:
  Взрослый человек играет в более зрелые игры (консоль, большой монитор).
  Читает книги — серьёзная литература, нон-фикшн.
  Сидит в машине — слушает музыку через колонки автомобиля.
  Идёт в кинотеатр — смотрит фильм на большом экране.
  → Карточки: «The Last of Us», «Sapiens», «Interstellar», «Hans Zimmer».

Сцена 4 — Зрелость:
  Пожилой человек в уютном кресле играет в спокойную игру (шахматы/головоломка).
  Читает книгу в очках, классика.
  Слушает музыку на виниловом проигрывателе или радио.
  Смотрит фильм дома на большом экране с семьёй.
  → Карточки: «Шахматы», «Война и мир», «Фрэнк Синатра», «Форрест Гамп».

Переход к таймлайну:
  После анимации пользователь начинает скроллить вниз.
  Появляется вертикальная хронологическая лента — от года 0 (рождения) до текущего возраста.
  При каждом скролле лента сдвигается по годам.
  Плавающие карточки фильмов, книг, игр и музыки скользят по этой ленте,
  появляясь и исчезая в зависимости от периода.
  Это создаёт ощущение «пролистывания жизни» — именно то, что предлагает TLOM.

В конце ленты — призыв к действию:
  «Начни свою историю»
  🔵 «Войти через Google»
  ✉️ «Зарегистрироваться по Email»

Техническая реализация лендинга:
  — React SPA, раздаётся через AWS CloudFront (CDN) с минимальной задержкой.
  — Анимации через CSS/JS анимации (Framer Motion или GSAP + ScrollTrigger).
  — Scroll-driven animation для хронологической ленты.
  — Никаких запросов к API — страница загружается мгновенно.
  — Оптимизация: lazy-loading для анимационных ресурсов, WebP/AVIF для изображений.

=== Этап 2: Регистрация и Вход ===

Сценарий A: Google OAuth 2.0 (один клик)
  1. Пользователь нажимает «Войти через Google»
  2. Открывается окно Google — выбирает аккаунт
  3. Google возвращает токен → бэкенд получает email, имя, аватарку
  4. Система проверяет: есть ли Account с таким GoogleId?
     — Нет → создаётся новый Account + пустой UserProfile
     — Да → просто входит
  5. Бэкенд генерирует JWT Access Token (15 мин) + Refresh Token (7 дней)
  6. Пользователь попадает на Dashboard

Сценарий B: Email + Пароль
  1. Пользователь вводит email, пароль, подтверждение пароля
  2. Бэкенд создаёт Account с PasswordHash (bcrypt)
  3. На email отправляется письмо для подтверждения (EmailConfirmed = false)
  4. После подтверждения → пользователь может войти
  5. JWT-токены → Dashboard

Под капотом:
  — Account хранится в AWS RDS (SQL Server)
  — Пароли хешируются, никогда не хранятся в открытом виде
  — Refresh Token хранится в БД с RefreshTokenExpiryTime
  — При каждом входе обновляется LastLoginAt

=== Этап 3: Первый вход — Заполнение профиля ===

Модальное окно или страница с приветствием: «Добро пожаловать! Расскажи немного о себе»

Поля:
  📸 Аватарка (загрузка файла)
  ✏️ Username (уникальный ник, например @cinephile_alex)
  📝 Bio (короткое описание)
  🎂 Дата рождения
  📍 Город / Страна

Под капотом:
  — Аватарка загружается в AWS S3 через presigned URL
  — В UserProfile.AvatarUrl сохраняется ссылка на S3/CloudFront
  — Username проверяется на уникальность в БД
  — Создаётся запись в AuditLog: Action: Created, EntityType: UserProfile

=== Этап 4: Dashboard — Главная страница пользователя ===

Что видит пользователь:
  — Краткая статистика: 🎬 12 фильмов | 📚 8 книг | 🎮 5 игр | 🎵 34 трека
  — Последние действия (свои): «Завершил The Last of Us Part II — 2ч назад»
  — Лента подписок: «@maria завершила Ведьмак 3 ⭐ 10/10»
  — Кнопка [+ Добавить контент]

  Под капотом:
  — Dashboard загружается через несколько параллельных API-запросов:
    GET /api/v1/entries?userId=me&sort=updatedAt&limit=5 — последние действия
    GET /api/v1/feed?cursor=...&limit=10 — лента подписок
    GET /api/v1/stats/summary — краткая статистика
    GET /api/v1/notifications?unreadOnly=true&limit=10 — непрочитанные уведомления
  — При загрузке Dashboard клиент подключается к SignalR Hub (/hubs/notifications)
    для получения real-time push-уведомлений
  — Данные кэшируются в Redis (ElastiCache) — повторные запросы мгновенны
  — Cursor-based пагинация для ленты подписок

=== Этап 5: Поиск и Добавление контента (Ключевой момент!) ===

Пользователь нажимает «+ Добавить контент»

Шаг 1: Выбор типа
  Четыре вкладки: 🎬 Фильмы | 📚 Книги | 🎮 Игры | 🎵 Музыка

Шаг 2: Поиск
  Пользователь выбирает «Фильмы» и вводит: «Interstellar»
  Бэкенд обращается к TMDB API → возвращает результаты
  Пользователь видит карточки с обложками и рейтингами из TMDB

Шаг 3: Добавление
  Пользователь нажимает «+ Добавить в таймлайн» на Interstellar (2014).

  Что происходит (логика общего пула MediaItem):
  1. Бэкенд получает: ExternalId = "tt0816692", ExternalSource = "TMDB"
  2. Проверка в БД: SELECT * FROM MediaItems WHERE ExternalId AND ExternalSource
  3a. Если НЕ найден (первый пользователь добавляет этот фильм):
      → Создаётся новый MediaItem из данных TMDB
      → Обложка скачивается → загружается в AWS S3
  3b. Если НАЙДЕН (кто-то уже добавлял):
      → Используется существующий MediaItem (никаких дублей!)
  4. Создаётся Entry: UserId = текущий, MediaItemId = найденный/созданный, Status = "Planned"
  5. Создаётся первый EntryEvent: Status = "Planned", DateTime = сейчас
  6. AuditLog: Action = "Created", EntityType = "Entry"

=== Этап 6: Работа с карточкой — Смена статусов ===

Пользователь открывает карточку Interstellar и видит:
  — Обложку, режиссёра, длительность, жанр, среднюю оценку
  — Свой текущий статус: [Planned]
  — Историю событий (EntryEvent)
  — Поля: оценка, ревью, потраченное время
  — Флажки: приватная запись, избранное

Смена статуса Planned → InProgress:
  1. PUT /api/v1/entries/{id}/status → { "status": "InProgress" }
  2. Обновляется Entry.Status = InProgress
  3. Создаётся новый EntryEvent: { Status: "InProgress", DateTime: now }
  4. В истории:
     • 28 апр 2026, 14:00 — Planned
     • 28 апр 2026, 20:30 — InProgress

Смена статуса InProgress → Completed:
  5. Добавляется ещё один EntryEvent
  6. Пользователь ставит оценку 9/10, пишет ревью
  7. В истории полная хронология:
     • 28 апр 2026, 14:00 — Planned
     • 28 апр 2026, 20:30 — InProgress
     • 28 апр 2026, 23:15 — Completed ✅

Полная хронология сохраняется навсегда — именно она формирует детальный таймлайн.

=== Этап 7: Визуальный таймлайн (Сердце проекта!) ===

Пользователь переходит на «Мой таймлайн» и видит:
  Горизонтальную или вертикальную ленту времени с карточками, расположенными
  по годам взаимодействия. Каждая карточка — это Entry с обложкой, оценкой и статусом.

  2024                    2025                    2026
  ──┬───────────────────────┬───────────────────────┬──
    │  📚 "Дюна" 9/10      │  🎮 "Ведьмак 3" 10/10│  🎬 "Interstellar" 9/10
    │  🎵 "Bohemian" 8/10  │  📚 "1984" 7/10      │
  ──┴───────────────────────┴───────────────────────┴──

Фильтрация и сортировка:
  — По типу: 🎬 Фильмы | 📚 Книги | 🎮 Игры | 🎵 Музыка | Все
  — По статусу: Completed / InProgress / Dropped / All
  — По периоду: конкретный год, диапазон, всё время
  — Сортировка: по дате, по рейтингу, по названию

Под капотом:
  GET /api/v1/timeline?userId=me&mediaType=all&status=completed
    &dateFrom=2024-01-01&dateTo=2026-12-31&sort=date_desc
    &cursor=eyJpZCI6MTIzfQ==&limit=20
  — Cursor-based пагинация — при скролле подгружаются следующие карточки
  — Данные из Redis если не изменились, иначе из RDS

=== Этап 8: Публичный профиль и Социальное взаимодействие ===

Публичный профиль: https://tlom.app/@cinephile_alex
  — Аватар, имя, Bio, город
  — Счётчики: 47 фильмов • 23 книги • 12 игр
  — Кнопка [Подписаться] + количество подписчиков
  — Публичный таймлайн (приватные записи скрыты)

Подписка (Follow):
  — @maria нажимает «Подписаться» на @cinephile_alex
  — Создаётся Follow: FollowerId = @maria, FollowingId = @cinephile_alex
  — Теперь в ленте @maria появляются действия @cinephile_alex

Лайки и Комментарии (на публичных карточках):
  — ❤️ Лайк — один на запись, можно убрать
  — 💬 Комментарий — текст под карточкой другого пользователя
  — Ограничения: нельзя лайкнуть свою запись, один лайк на один Entry
  — При каждом действии (Follow/Like/Comment) создаётся Notification
    и отправляется через SignalR владельцу записи/профиля

=== Этап 8.5: Уведомления (Notifications) ===

Пользователь видит иконку 🔔 в хедере с счётчиком непрочитанных.
При клике открывается дропдаун/страница со списком:
  — «@maria подписалась на вас» — 2 мин назад
  — «@alex лайкнул вашу запись Interstellar» — 1ч назад
  — «Добро пожаловать в TLOM!» — сегодня

Действия:
  — Клик на уведомление → переход к связанной сущности + IsRead = true
  — Кнопка «Пометить всё как прочитанное»

Для админа:
  — Отдельный поток уведомлений с админ-алертами
  — «Пользователь @alex удалил запись #456»
  — «Новый пользователь @john зарегистрировался»

Под капотом:
  — SignalR Hub: /hubs/notifications (авторизованный, JWT-токен в query string)
  — GET /api/v1/notifications?cursor=...&limit=20 — история уведомлений
  — PUT /api/v1/notifications/{id}/read — пометить как прочитанное
  — PUT /api/v1/notifications/read-all — пометить всё
  — Notification сохраняется в RDS + push через SignalR одновременно

Пользователь переходит в «Моя статистика» и видит:
  — Годовой отчёт: «В 2025 ты прочитал 23 книги, прошёл 8 игр, посмотрел 47 фильмов»
  — Общее время: ~340 часов
  — Любимый жанр: Sci-Fi (34%)
  — Средняя оценка: 7.8 / 10
  — Самый продуктивный месяц: Август (42 шт)
  — Топ оценки года: 1. Ведьмак 3 — 10/10, 2. Interstellar — 9/10
  — Тренд по годам: гистограмма 2023→2024→2025 с ростом
  — Кнопка [Поделиться статистикой]

Под капотом:
  — GET /api/v1/stats/yearly?year=2025
  — Статистика вычисляется из Entry + EntryEvent + MediaItem
  — Кэшируется в Redis (TTL: 1 час)
  — Тяжёлые агрегации можно выносить в фоновые задачи

=== Этап 10: Админ-панель ===

Администратор входит через тот же логин, но с Role = Admin.

Функции:
  — Аудит-лог: таблица с фильтрацией (юзер, действие, дата, тип сущности)
    Каждая запись: Время | Юзер | Действие (Created/Updated/Deleted) | Тип сущности
    OldValues / NewValues хранятся как JSON-снимки
  — Управление пользователями: просмотр карточек, редактирование, удаление, блокировка
  — Системная статистика: всего пользователей, карточек за сегодня, активных сессий

Под капотом:
  — Доступ: [Authorize(Roles = "Admin")]
  — Offset-пагинация для табличных данных
  — Фильтрация через query-string

=== Техническая схема: AWS инфраструктура ===

  Пользователь (Браузер)
         │
         ▼
  CloudFront (CDN) — раздаёт статику
         │
    ┌────┴────┐
    ▼         ▼
  S3 Bucket   ECS Fargate
  (статика +  (ASP.NET Core API)
   аватарки)       │
              ┌────┴────┐
              ▼         ▼
         RDS         ElastiCache
      (SQL Server)    (Redis)
       — Accounts     — Кэш
       — Profiles     — Сессии
       — Entries
       — MediaItems
       — Notifications
       — AuditLogs

  SignalR Hub: /hubs/notifications (через ECS Fargate, WebSocket)

  Внешние API: TMDB | Google Books API | Steam/CheapShark | Spotify API

=== Жизненный цикл одного API-запроса ===

  Фронтенд (React)
    POST /api/v1/entries { mediaItemId: 123, status: "Planned" }
    Header: Authorization: Bearer eyJhbGci...
         │
         ▼
  ASP.NET Core Pipeline (Middleware Stack):

  1. Rate Limiting Middleware → проверка лимита запросов
  2. Authentication Middleware (JWT) → валидация токена, извлечение UserId
  3. ExceptionHandlingMiddleware → оборачивает всё в try/catch → ProblemDetails (RFC 7807)
  4. Controller → десериализация → CreateEntryCommand → MediatR
  5. MediatR Pipeline:
     a) ValidationBehavior → FluentValidation проверяет поля → если ошибка → 400
     b) CreateEntryCommandHandler:
        — Бизнес-логика: поиск/создание MediaItem по ExternalId
        — Создание Entry + EntryEvent
        — AutoMapper: Entity → ResponseDTO
        — Запись в RDS через EF Core
        — Инвалидация кэша в Redis
        — Запись AuditLog
        — Отправка Notification через INotificationService (БД + SignalR push)
  6. Response: 201 Created { "id": 456, "status": "Planned", ... }
